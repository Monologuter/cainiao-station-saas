## 2. 执行总原则

1. **演进式架构 / 边界先行**：按限界上下文分 Nest 模块，只走接口 + 事件，不串库。单体部署，可拆服务。
2. **适配层先跑 mock**：短信/支付/物流/OCR 在 P1–P3 全用降级实现，P4（P4-4）再接真，业务代码不动。
3. **TDD（红→绿→重构）+ 频繁提交**：每 Task 一次 commit（约定式 message）。
4. **多租户安全是隐性验收项**：每新增业务表必须加 `tenant_id` + RLS Policy + `FORCE ROW LEVEL SECURITY`（消费者只读通道除外）。
5. **每个 plan 交付即可跑可测**，不留半成品；后端 plan 收尾有 e2e 冒烟，前端 plan 收尾有可点通流程。
6. **执行编排**：用 `superpowers:subagent-driven-development`（每 Task 派新子代理 + 两阶段评审）。

## 4. 单个 Plan 的标准执行流程（SOP）

对队列里任意一个 plan，执行四步：

**① 展开**（仅 P1-2 起的设计级文档需要；P1-1 已是代码级，跳过）
用 `superpowers:writing-plans` 或子代理，把该 plan 的设计文档展开为**逐步 TDD 代码级计划**，粒度对齐 `P1-1-后端地基.md`（每 Task → 写失败测试/最小实现/跑验证/提交）。产出 `plans/<plan-id>-<名>-执行.md`（或就地补充）。

**② 执行**
用 `superpowers:subagent-driven-development`：按 Task 顺序，每 Task 派一个新子代理实现 → 两阶段评审（自检 + 复核）→ 通过进入下一 Task。每 Task 结束 commit。

**③ 验收**
跑该 plan 的验收（后端：e2e 冒烟串本期闭环；前端：可点通页面流程）；对照该 plan「验收标准」逐条核对；确认新表 RLS 已加。

**④ 收尾**
回到本总控，更新进度清单状态；若触发期末 Gate（见 §5），跑分期验收。

---

## 5. 分期验收门（Gate）

每期全部 plan 完成后过一次 Gate，绿灯才进下一期：

- **P1 Gate（MVP 可用）**：平台开店 → 店长登录(RBAC) → 扫码入库→自动通知→取件核销→库位释放 全流程跑通；用户端按手机号查件并出示取件码；基础统计有数。RLS 跨租户隔离 e2e 通过。
- **P2 Gate（经营闭环）**：寄件下单→模拟支付→模拟轨迹追踪；滞留定时扫描+催取；异常件工单流转；积分到账；运营大屏实时刷新。
- **P3 Gate（可商业化）**：套餐订阅+月费/用量混合出账+欠费停用；自助入驻申请→审核→开通；平台多门店监控+审计+渠道开关；限流/缓存/可观测就位。
- **P4 Gate（智能 + 生产就绪）**：OCR 拍照入库；大模型客服可答本人包裹；智能库位/预测上线（可降级）；短信/支付/物流/小程序消息切真实并回归通过。
