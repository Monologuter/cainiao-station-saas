# P3-4 工程化加固 - 执行级 TDD 计划

> 来源：`P3-4-工程化加固.md`。本文将设计级加固任务展开为逐步 TDD 队列，按「先观测、再治理、再验收」推进。

## 执行范围

- 后端横切层：`core/observability`、`core/rate-limit`、`core/circuit-breaker`、`core/cache`、`core/lock`、`core/scheduler-lock`、`core/idempotency`、`core/security`。
- 业务接入点：identity 登录、pickup 核销、inbound 入库、shipping/pay 支付、notify/pay/logistics 适配层、P2/P3 定时任务。
- 验收：`test/hardening.e2e-spec.ts` 覆盖限流、熔断、双实例防重、健康检查、指标、缓存写时失效。

## 约束补充

- 横切能力通过装饰器、拦截器、守卫、注入式服务接入，避免改写领域逻辑。
- Redis 不可用策略必须按场景显式：安全路径 fail-closed，普通读缓存 fail-open 回源 DB。
- 新平台级表（如 `scheduled_locks`）不带 `tenant_id`，必须写表注释说明平台级锁表、显式 RLS 例外。
- 指标/日志不得泄露敏感字段；traceId 可贯穿日志、响应头、审计，但不进入业务主键。
- 每个任务完成后独立 commit；涉及全局拦截器/守卫的任务必须跑后端全量 unit/e2e/build。

## Task 1：可观测性地基

### RED
- 单测覆盖 trace 上下文：
  - 入站无 `X-Request-Id` 时生成 traceId。
  - 入站有 `X-Request-Id` 时复用并写回响应头。
  - `TenantContext` 中可读取 traceId。
- 集成/e2e 覆盖：
  - `/health/live` 返回存活。
  - `/health/ready` 检查 PostgreSQL 与 Redis。
  - `/metrics` 返回 Prometheus 文本，包含 HTTP 请求计数/耗时指标。

### GREEN
- 扩展 `TenantContext` store：`traceId`。
- 新增 `core/observability`：trace interceptor/middleware、metrics service/interceptor、health controller。
- 全局接入 trace 与 HTTP metrics。

### 验收
- `npm test -- --runInBand observability` 相关单测绿。
- `npm run test:e2e -- --runInBand health-metrics.e2e-spec.ts` 绿。
- Commit：`feat(backend): add observability foundation`

## Task 2：分布式锁工具

### RED
- 单测覆盖：
  - 多个并发请求同一 key 仅一个拿到锁。
  - 释放时 token 不匹配不得误删他人锁。
  - TTL 到期后可重新获取。
  - Redis 不可用时强一致路径返回业务错误。

### GREEN
- 新增 `core/lock`：`RedisLockService` 标准化 `acquire/runWithLock/release/extend`。
- 统一锁 key 规范与 token 生成。

### 验收
- 锁单测绿。
- Commit：`feat(backend): add distributed lock primitive`

## Task 3：定时任务多实例防重

### RED
- 迁移/模型测试覆盖：
  - `scheduled_locks` 表存在且带注释说明平台级 RLS 例外。
  - 唯一 key 为 `name`。
- 集成测覆盖：
  - 两个并发执行同一 `name`，只有一个进入任务体。
  - 被锁跳过时返回 skipped 并记录指标。

### GREEN
- Prisma model + migration：`ScheduledLock`。
- 新增 `core/scheduler-lock`：`ScheduledLockService`、`@DistributedCron` 辅助包装。
- 接入滞留扫描、出账、报表任务的执行入口。

### 验收
- `scheduler-lock` 单测/集成测绿。
- Commit：`feat(backend): guard scheduled jobs with locks`

## Task 4：接口限流

### RED
- 单测覆盖 Redis token bucket 与 sliding window：
  - 桶容量耗尽返回 retryAfter。
  - 时间推进后令牌补充。
  - 滑动窗口超过阈值拒绝。
- e2e 覆盖：
  - 登录超阈值返回限流业务码和 Retry-After。
  - 核销/入库标注接口走令牌桶。

### GREEN
- 新增 `ApiCode.RATE_LIMITED`。
- 新增 `core/rate-limit`：`@RateLimit`、`RateLimitGuard`、Redis Lua/降级内存实现、响应头。
- 接入登录、取件核销、入库、寄件支付。

### 验收
- 限流 unit/e2e 绿。
- Commit：`feat(backend): add redis rate limiting`

## Task 5：适配层熔断

### RED
- 单测覆盖熔断状态机：
  - CLOSED 失败率达阈值进入 OPEN。
  - OPEN 直接 short-circuit 走 fallback。
  - coolDown 后 HALF_OPEN，成功恢复 CLOSED，失败回 OPEN。
  - 超时计为失败。

### GREEN
- 新增 `core/circuit-breaker`：`CircuitBreakerService`、状态指标。
- 接入 notify/pay/logistics adapter 外呼；P4 OCR 预留 channel。

### 验收
- 熔断单测绿，notify/pay/logistics 既有测试仍绿。
- Commit：`feat(backend): add adapter circuit breaker`

## Task 6：多级缓存

### RED
- 单测覆盖：
  - L1 命中不访问 Redis/DB。
  - L2 命中回填 L1。
  - DB 回源回填 L2/L1。
  - 空值短 TTL 防穿透。
  - 写时失效删除 L2 并广播 L1 失效。

### GREEN
- 新增 `core/cache`：L1 LRU、Redis L2、key builder、TTL jitter、single-flight rebuild、invalidator。
- 接入字典/渠道开关读取，随后接权限/取件码热点。

### 验收
- 缓存单测绿；字典/渠道 e2e 仍绿。
- Commit：`feat(backend): add multi-level cache`

## Task 7：并发与幂等收口

### RED
- 单测/e2e 覆盖：
  - 同一幂等键重复写请求只执行一次并返回同一结果。
  - 并发核销只成功一次。
  - 重复 EventBus 事件只被同一 handler 生效一次。

### GREEN
- 新增 `core/idempotency`：请求幂等记录、事件 handler 去重。
- 将 pickup/pay/关键 EventBus subscriber 接入幂等辅助。
- 文档化乐观锁和分布式锁组合规范。

### 验收
- 幂等与并发测试绿。
- Commit：`feat(backend): standardize idempotency guards`

## Task 8：安全加固

### RED
- 单测/e2e 覆盖：
  - refresh token 签发、轮换、吊销。
  - refresh token 复用检测后吊销用户全部会话。
  - 改密后旧 refresh 不可用。
  - 响应/日志敏感字段脱敏。

### GREEN
- identity 新增 refresh token 机制与 Redis session store。
- 登录返回 access + refresh；新增 refresh/logout/change-password 端点。
- 脱敏 serializer/响应工具复用 audit redaction。

### 验收
- identity 安全测试绿；既有 station/admin/user 登录流兼容。
- Commit：`feat(backend): harden auth sessions`
