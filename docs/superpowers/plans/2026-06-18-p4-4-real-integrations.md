## 1. 目标 · 周期 · 依赖

### 1.1 目标

把前期跑 mock/降级的四类适配器切换为**真实外部服务**，且满足三条硬约束：

1. **接口不变、业务零改动**：`NotifyChannel / PayChannel / LogisticsProvider` 接口在 P1–P3 已定型，本期只新增 `real` 实现类，不动调用方（`notify/pay/logistics/shipping` 业务代码）。
2. **配置开关切换**：每个集成点一个 provider 开关（如 `NOTIFY_SMS_PROVIDER=mock|tencent`），运行时由 P3-3 渠道开关读取选择注入。
3. **随时可回降级**：任何真实服务故障或资质未就绪，开关切回 `mock` 即恢复，核心代收/代寄闭环不被外部依赖卡死。

四类目标服务：

| 集成点 | 接口 | 降级（现状） | 真实（本期接入） | 开关 |
|---|---|---|---|---|
| 短信 | `NotifyChannel` | 控制台模拟短信 | **腾讯云短信** | `NOTIFY_SMS_PROVIDER=mock\|tencent` |
| 微信通知 | `NotifyChannel` | 站内/H5 | **微信小程序订阅消息** | `NOTIFY_WECHAT_PROVIDER=mock\|wechat` |
| 支付 | `PayChannel` | 模拟支付 | **微信支付（JSAPI/Native）** | `PAY_PROVIDER=mock\|wechat` |
| 物流 | `LogisticsProvider` | 模拟轨迹推进 | **快递100 或 快递鸟** | `LOGISTICS_PROVIDER=mock\|kuaidi100\|kdniao` |

### 1.2 周期

本期内部**四个 provider 可并行接入**（互相解耦，只共用开关/密钥/熔断三类横切设施），建议按「先短信 → 小程序订阅 → 物流 → 支付」的资质就绪顺序滚动上线（支付资质周期最长，见 §8）。整体投入估算见 §9 任务分解。

### 1.3 依赖

- **P2-1**：`PayChannel / LogisticsProvider` 接口与 mock 实现、`payments / logistics_tracks` 表、`ShipOrderPaid` 事件、`ship-order` 状态机。
- **P1-2**：`NotifyChannel` 接口、`InAppChannel / MockSmsChannel`、`notifications / notify_templates` 表、`ParcelStored` 触发链。
- **P3-4**：适配层熔断器、可观测性（traceId/指标/日志）、密钥不入日志的脱敏，是真实外呼的稳健性前提。
- **P3-3**：`config` 渠道开关持久化 + 热生效缓存，是 provider 选择的运行时来源。
- **P3-1**：用量计量（`usage.service` / `usage_records`），真实短信条数回流计费靠它。

## 2. 切换原则（贯穿全期的设计约束）

1. **接口即契约，不为真实实现改契约**：若真实服务需要 mock 没有的入参/出参（如微信支付的 `prepay_id`、物流的 `nu+com`），通过**接口已有的上下文对象/扩展字段**承载，不破坏既有方法签名；确需扩展时向后兼容（可选字段）。
2. **新增 real 实现，不改 mock**：每个集成点目录 `providers/` 下并存 `*.mock.ts` 与 `*.real.ts`（或 `*.tencent.ts` / `*.wechat.ts` / `*.kuaidi100.ts`），共同 `implements` 同一接口。mock 保留为永久兜底，不删除。
3. **配置开关 + 工厂注入**：每个集成点一个 `*ChannelFactory / *ProviderFactory`，读 provider 开关返回对应实现；NestJS 用动态 provider（`useFactory`）按 env/config 注入。开关来源优先级：DB 渠道开关（P3-3，热生效）→ 环境变量（默认/兜底）。
4. **灰度可控**：开关支持**按租户/门店维度**覆盖（默认全局值，特定租户可单独切 real），先小范围灰度真实通道，再放量。开关变更走 P3-3 审计。
5. **失败自动回降级 + 可手动回退**：
   - 运行时：真实 Provider 外呼失败 → 经 P3-4 熔断器，熔断打开时**自动降级**到 mock（或站内通知兜底），保证不丢消息/不阻断业务。
   - 运维侧：开关切回 `mock` 即整体回退，无需发版。
6. **真实路径全程可观测**：每次外呼记 traceId、目标 provider、耗时、结果码、第三方 requestId；回调/对账差异落审计与告警。
7. **测试不打真实生产**：单测打桩第三方 SDK；集成测走第三方**沙箱/测试号**或本地桩服务；CI 默认 `mock`，真实通道走专用集成环境。

## 3. 腾讯云短信（TencentSmsChannel）

### 3.1 配置与资质

- **SDK**：`tencentcloud-sdk-nodejs`（sms 模块）。
- **配置项**（密钥走 §7 密钥管理，不入代码/日志）：
  - `TENCENT_SMS_SECRET_ID` / `TENCENT_SMS_SECRET_KEY`
  - `TENCENT_SMS_SDK_APP_ID`（短信应用 ID）
  - `TENCENT_SMS_SIGN_NAME`（已报备签名）
  - `TENCENT_SMS_REGION`（如 `ap-guangzhou`）
- **资质**：签名 + 模板需在腾讯云控制台报备审核通过（见 §8 周期）。

### 3.2 签名与模板映射

- 本系统的 `notify_templates`（业务模板键，如 `PARCEL_STORED / OVERDUE_L1 / OVERDUE_L2 / OVERDUE_FINAL`）与腾讯云**已报备模板 ID**建立映射表（配置化，存 `notify_templates` 扩展字段或 `config` 字典），切 provider 时不改业务模板键。
- 业务侧只传**模板键 + 变量数组**（如 `[取件码, 货架位]`）；`TencentSmsChannel` 负责：业务模板键 → 腾讯模板 ID、变量数组顺序对齐腾讯模板占位符 `{1}{2}`、套用签名名。
- 多租户：签名归属。MVP 用平台统一签名；为支持租户自有签名预留 `tenant → signName/templateId` 覆盖映射（按租户报备）。

### 3.3 计量上报（billing 衔接）

- 发送**成功**后产出用量事件（如 `SmsSent { tenantId, count, provider, bizTemplate }`），由 P3-1 `usage.service` 累加 `usage_records`（计量项=短信条数，按订阅/账期幂等）。
- 计量以**真实计费条数**为准：长短信按腾讯返回的拆分条数计（一条超 70 字按多条计），从 SDK 返回的 `SendStatusSet` 取每号码实际计费条数。
- 只有 mock 不产生真实费用：mock 通道**不上报计量**（或标记 `provider=mock` 不计费），避免污染账单。

### 3.4 失败处理与重试

- **同步返回错误分类**：
  - 可重试（限流 `RequestLimitExceeded`、网络超时、5xx）：交 BullMQ notify 队列**指数退避重试**（复用 P1-2 通知异步链路），最大重试次数可配。
  - 不可重试（签名/模板未审核、欠费、号码格式非法、内容违规）：不重试，落 `notifications` 失败态 + 告警，避免无效重试烧配额。
- **熔断**：连续失败率/超时超阈 → P3-4 熔断打开 → 自动降级（站内通知兜底，或回 mock 控制台），半开探测恢复。
- **幂等**：同一通知任务带幂等键（`notification.id`），重试不产生重复短信（依赖队列去重 + 发送前查状态）。

### 3.5 回归口径

- 单测：模板键映射、变量顺序、计费条数解析、错误分类（可重试/不可重试）、计量事件产出，全部打桩 SDK。
- 集成测：开关 `tencent` 走沙箱/测试号真发一条 `PARCEL_STORED`，断言 `notifications` 成功态 + `usage_records` +1 + traceId 落库；关闭网络模拟失败断言回降级。

## 4. 微信支付（WechatPayChannel）

### 4.1 配置与资质

- **SDK**：`wechatpay-node-v3`（API v3）。
- **配置项**：`WXPAY_MCH_ID`（商户号）、`WXPAY_APP_ID`、`WXPAY_API_V3_KEY`（APIv3 密钥）、商户**私钥**与**证书序列号**、`WXPAY_NOTIFY_URL`（回调地址）、`WXPAY_CERT_DIR`（平台证书目录）。
- **资质**：微信支付商户号开通 + 与小程序/公众号 AppID 绑定 + 域名/回调 ICP 备案（见 §8，周期最长）。

### 4.2 统一下单（支付方式）

- 主路径 **JSAPI**（user-app 小程序内支付）；H5/桌面备 **Native（扫码）**。`PayChannel.createPayment(order)` 内部按场景选下单接口，返回前端发起支付所需参数（JSAPI：`prepay_id` 二次签名包；Native：`code_url` 二维码）。
- 下单入参由 `ship-order` + `payments` 提供：商户订单号 `out_trade_no`（= payment 记录的稳定唯一键，幂等防重复下单）、金额（分）、商品描述、用户 `openid`（JSAPI 必需，由 user-app 登录态提供）。
- 金额、订单号一律以**本系统 payment 记录**为权威，第三方仅用于发起与确认，不信任前端回传金额。

### 4.3 回调验签与状态推进

- **回调验签**（安全核心，见 §11）：用微信平台证书验 `Wechatpay-Signature` 签名 + 校验时间戳/Nonce 防重放 + APIv3 密钥解密回调密文，三步缺一不可，验签失败直接拒绝并告警。
- **幂等**：回调以 `out_trade_no` 查 `payments`；已是 `PAID` 直接返回成功（微信会重试回调），不重复推进、不重复发 `ShipOrderPaid`。
- **状态衔接**：验签通过 + 金额一致 + 交易成功 → 更新 `payments` 为成功 + 落第三方 `transaction_id` → 推进 `ship-order` `CREATED→PAID` → 发 `ShipOrderPaid`（下游揽收/积分/计量复用 P2-1/P2-3 链路，零改动）。
- **主动查单兜底**：回调久未到，定时**主动查单**（`out_trade_no` 查微信支付状态）补偿确认，防回调丢失导致订单悬挂。

### 4.4 退款

- `PayChannel.refund(payment, amount, reason)`：调微信退款 API，商户退款单号幂等键；退款回调同样验签 + 幂等；退款成功更新 `payments` 退款态，必要时联动 `ship-order` 取消/退款态。
- 退款金额受原支付金额约束（不超额），多次部分退款累计校验。

### 4.5 对账（billing 衔接）

- **每日对账**定时任务（BullMQ，单实例防重）：拉微信支付**对账单**，与本系统 `payments` 逐笔比对（金额/状态/笔数）。
- 差异分类：本系统有微信无（疑似下单未支付）、微信有本系统无（回调丢失，触发查单补偿）、金额不符（人工核查告警）。差异落 `audit_logs` + 告警，不自动改账。
- 与 billing 衔接：寄件抽佣/支付流水进 P3-1 计费视角（若计），以对账后数据为准。

### 4.6 回归口径

- 单测：下单参数组装、回调验签（伪造签名拒绝）、金额一致性校验、幂等（重复回调只推进一次）、退款金额约束，全打桩 SDK。
- 集成测：沙箱/打桩走「下单→回调→`payments`成功→`ship-order` PAID→`ShipOrderPaid`」闭环；伪造回调断言被拒；模拟回调丢失断言查单补偿补齐。

## 5. 物流轨迹（LogisticsProvider · 快递100 / 快递鸟）

### 5.1 选型

- **快递100** 与 **快递鸟** 二选一作主，另一为备（接口抽象一致，实现可切）。选型维度：覆盖快递商范围、订阅推送 vs 轮询能力、计费、对接成本。
- **默认推荐快递100**（订阅推送成熟、文档清晰）；保留 `KdNiaoProvider` 实现以便切换/灰度。开关 `LOGISTICS_PROVIDER=kuaidi100|kdniao|mock`。

### 5.2 订阅推送 vs 轮询

- **优先订阅推送（push）**：下单/揽收后向快递100/快递鸟订阅该运单（`com 快递公司编码 + nu 运单号 +（顺丰需手机后四位）`），第三方有轨迹更新时**回调**本系统推送接口，写入 `logistics_tracks`。低延迟、省配额。
- **轮询兜底**：对未支持推送或推送异常的运单，定时（BullMQ）拉取轨迹补偿；已签收/已终态的运单停止轮询。
- 两种模式共用同一**轨迹入库与映射逻辑**，仅数据来源不同。

### 5.3 映射到 logistics_tracks

- 第三方轨迹结构（时间、节点描述、状态码）→ 标准化映射到 `logistics_tracks`（运单/节点时间/描述/标准状态）。
- 快递商状态码 → 本系统 `ship-order` 状态映射：揽收→`COLLECTED`、运输中→`IN_TRANSIT`、派送/签收→`DELIVERED`，推进 `ship-order` 状态机（复用 P2-1）。
- **去重幂等**：同运单同节点（时间+描述指纹）只入一次，推送+轮询重复到达不产生重复轨迹。

### 5.4 异常预警

- 识别异常状态（退回、疑难、超时未更新、拒收）→ 标记 + 发预警事件（接 P2-2 异常工单/通知或 analytics 看板）。
- 长时间无轨迹更新（超阈）触发轮询探测 + 运营提醒。

### 5.5 回归口径

- 单测：轨迹映射、状态码→`ship-order`状态映射、节点去重指纹、异常识别。
- 集成测：开关 `kuaidi100` 模拟推送回调写入 `logistics_tracks` 并推进状态；模拟推送失败走轮询补偿；伪造推送验签（若提供 sign）被拒。失败回 `mock` 模拟轨迹保证寄件追踪不中断。

## 6. 微信小程序订阅消息（WechatSubscribeChannel）

### 6.1 模板与下发

- 在微信公众平台申请**订阅消息模板**（如「快递到件提醒」），模板字段映射本系统通知数据（取件码、货架位、门店名）。本系统 `notify_templates` 业务键 → 微信模板 ID + 字段映射。
- 下发用微信 `subscribeMessage.send`：需 `touser`（用户 openid）、`template_id`、`data`、`page`（点击跳转小程序页，如取件页）。

### 6.2 用户授权态处理（一次性订阅约束）

- 微信订阅消息是**一次性授权**：用户每次授权只能下发一条。设计上：
  - user-app 在关键操作（绑定手机、查件）引导用户**点击订阅**，记录授权额度。
  - 下发前检查该 openid 对该模板的**剩余授权次数**，无授权则**不下发并回降级**（短信/站内），不报错阻断。
  - 授权额度与 openid 绑定，存于 `consumers`/通知授权表（按 openid+templateId 计数）。

### 6.3 多渠道并存策略（站内 / 短信 / 小程序订阅）

- `notify.service` 是统一出口，单条通知按**渠道优先级 + 可用性**择路，而非各渠道各发一遍（避免重复打扰、控制短信成本）：
  - 优先级（可配，按通知类型）：**小程序订阅（有授权且免费）> 站内（always）> 短信（兜底/重要）**。
  - 重要通知（如最终催取）可**多通道叠加**（订阅 + 短信）；普通通知优先免费通道。
- 渠道选择失败链：订阅无授权/失败 → 短信；短信无配额/失败 → 站内必达。站内永远兜底。
- 各渠道发送结果统一落 `notifications`（带 channel 字段），便于审计与触达率统计。

### 6.4 回归口径

- 单测：模板字段映射、授权额度判断（有/无授权分支）、渠道择路（优先级 + 降级链）。
- 集成测：开关 `wechat` 打桩下发订阅消息；无授权断言回短信/站内；多通道叠加断言各渠道各一条 `notifications`。

## 7. 配置与开关设计

### 7.1 Provider 开关

| 开关 | 取值 | 作用域 | 默认 |
|---|---|---|---|
| `NOTIFY_SMS_PROVIDER` | `mock` / `tencent` | 全局 + 租户可覆盖 | `mock` |
| `NOTIFY_WECHAT_PROVIDER` | `mock` / `wechat` | 全局 + 租户可覆盖 | `mock` |
| `PAY_PROVIDER` | `mock` / `wechat` | 全局 + 租户可覆盖 | `mock` |
| `LOGISTICS_PROVIDER` | `mock` / `kuaidi100` / `kdniao` | 全局 + 租户可覆盖 | `mock` |

- **来源优先级**：DB 渠道开关（P3-3 `system_configs`，热生效、可灰度按租户覆盖）→ 环境变量兜底默认。
- **注入机制**：各集成点工厂（`useFactory`）按开关返回实现实例；开关变更后下次请求生效（热生效缓存失效），无需重启。

### 7.2 密钥管理

- 所有第三方密钥（SecretKey、APIv3Key、商户私钥、各 API Key）**只存环境变量 / 密钥管理（如 Docker secret / 部署侧 vault）**，不入代码库、不入 DB 明文、不入日志（脱敏复用 P3-4）。
- 证书类（微信商户私钥/平台证书）挂载只读卷，权限最小化。
- 配置加载启动时校验：选了真实 provider 但密钥缺失 → 启动告警并**安全回落 mock**（不裸奔崩溃）。

### 7.3 环境分离

- **dev/test**：默认全 `mock`，CI 不触真实服务；需要时连第三方**沙箱/测试号**。
- **staging（集成环境）**：连真实沙箱或专用测试商户号/签名，跑真实通道回归。
- **prod**：真实生产凭证，按 §8 资质就绪后逐项灰度切 real。
- 回调地址（支付/物流推送）按环境分离配置，回调白名单按环境校验来源。

|---|---|---|---|

- **上线准备清单**（每项）：凭证就绪 → staging 沙箱回归绿 → 灰度单租户切 real → 监控对账无异常 → 放量。
- **关键路径**：微信支付资质周期最长，应**最先发起申请**、最后切换；短信/物流/订阅可先行。

## 10. 验收标准（各通道真实回归）

1. **零业务改动**：四类接入仅新增 `*.real`/`*.<provider>` 实现 + 工厂/开关，`notify/pay/logistics/shipping` 业务调用方代码与既有单测不变。
2. **短信真实**：开关 `tencent` 在沙箱/测试号真发成功，`notifications` 成功态、`usage_records` 计量 +1（真实条数）、traceId 落库；失败可重试项重试、不可重试项告警、熔断时回降级。
3. **微信支付真实**：沙箱/打桩跑通「下单→回调验签→`payments` 成功→`ship-order` PAID→`ShipOrderPaid`」；伪造回调被拒；重复回调幂等只推进一次；退款与对账任务可运行、差异可发现。
4. **物流真实**：开关 `kuaidi100`（或 `kdniao`）推送回调/轮询写入 `logistics_tracks` 并正确推进 `ship-order` 状态；节点去重；异常状态触发预警；失败回 `mock` 轨迹不中断追踪。
5. **小程序订阅真实**：开关 `wechat` 有授权时下发订阅消息；无授权回降级短信/站内；多渠道择路与降级链正确，站内必达。
6. **一键回降级**：任一集成点开关切回 `mock` 立即恢复降级实现，核心代收/代寄闭环全绿（e2e 验证）。
7. **回归全绿**：`test/real-integrations.e2e-spec.ts` 覆盖四通道 mock↔real 切换、真路径打桩/沙箱跑通、失败回降级。

## 11. 依赖与风险

| 风险 | 说明 | 缓解 |
|---|---|---|
| **资质周期** | 微信支付商户审核最长（数周），阻塞支付上线 | 最先发起申请；其余三项先行；未就绪保持 mock，业务不受影响 |
| **回调安全** | 支付/物流回调若验签缺失易被伪造刷单/改状态 | 强制验签（微信平台证书/APIv3 解密、防重放、来源白名单）；金额以本系统为权威；验签失败拒绝并告警（§4.3/§5.2/T0.3） |
| **回调可靠性** | 回调可能丢失/重复 | 幂等（`out_trade_no`/节点指纹）+ 主动查单/轮询补偿（§4.3/§5.2） |
| **对账差异** | 支付实际与本系统不符导致资损 | 每日对账任务、差异分类告警、不自动改账人工核查（§4.5） |
| **短信成本与频控** | 真实短信按条计费、易超额/扰民 | 多渠道优先免费通道（订阅/站内）、催取分级去重（P2-2）、计量回流计费可控（§3.3/§6.3） |
| **真实服务故障** | 第三方限流/宕机影响业务 | 熔断 + 自动降级（P3-4）+ 一键开关回 mock（§2.5/§7） |
| **密钥泄露** | 凭证/证书泄露 | 密钥仅环境变量/密钥管理、不入库不入日志、证书只读挂载、最小权限（§7.2） |
| **隐私合规** | 运单号/手机号/openid 等敏感数据 | 脱敏存储与日志、最小化使用、符合各服务数据合规要求（§8） |

**直接依赖**：P2-1（pay/logistics/notify 接口与降级实现及表）、P3-4（熔断/可观测/脱敏）、P3-3（渠道开关热生效与灰度）、P3-1（用量计量回流）、P1-2（notify 触发链与通知表）。
